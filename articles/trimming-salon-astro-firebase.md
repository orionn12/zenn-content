## はじめに

知人が営む犬猫専門トリミングサロン「わんにゃんサロンMOON」のWebサイトを制作しました。

最初は別ブログに相乗りする形で **WordPress** で作り始めたのですが、要件を詰めていくうちに方向性が変わり、最終的に **Astro + Firebase Hosting の静的サイト** に着地しました。

この記事では、

- なぜWordPressをやめて静的サイトにしたのか
- AstroとFirebaseでどう構成したのか
- デザインを「専用のWebデザインAI」から **Claude** に切り替えたら想像以上に良かった話
- お知らせ（Topics）を **Googleスプレッドシート** で非エンジニアでも更新できるようにした仕組み

を、実際のコードを交えて書いていきます。個人開発・小規模な店舗サイトを頼まれた人の参考になれば。

## 当初はWordPress、でも予約機能をやめて方針転換

スタート時点では、知人がすでに運用していたブログに相乗りする形で WordPress で構築する想定でした。「予約機能を持たせたい」という話があったので、動的なCMSであるWordPressは妥当な選択に見えました。

ところが要件を詰めていくと、

- 予約はこれまで通り電話・SNSのDMで十分回っている
- システム予約を入れると運用（ダブルブッキング対応・キャンセル処理など）の負担がむしろ増える

ということで、**予約機能は持たせない方向** で合意しました。

予約という動的要素がなくなると、サイトに残るのは「店舗紹介」「メニュー」「アクセス」「お知らせ」といった、ほぼ更新頻度の低い静的なコンテンツだけです。

それなら……

> 動的なものが無いなら、WordPressのサーバー代・保守コストを払い続ける理由がない

ということで、**静的サイト化して維持コストを削減する** 方針に切り替えました。WordPressはプラグインやコア本体のアップデート追従、セキュリティ対応も付いて回るので、身内向けの小規模サイトでそれを背負い続けるのは割に合いません。

## 技術スタック：Astro + Tailwind + Firebase Hosting

最終的な構成はこちらです。

| 項目 | 採用技術 |
| --- | --- |
| フレームワーク | Astro 6 |
| スタイリング | Tailwind CSS 4（`@tailwindcss/vite`） |
| ホスティング | Firebase Hosting |
| お知らせ管理 | Googleスプレッドシート |

`package.json` はとてもシンプルです。

```json
{
  "name": "wannyan-salon-moon",
  "type": "module",
  "dependencies": {
    "@tailwindcss/vite": "^4.2.4",
    "astro": "^6.1.8",
    "tailwindcss": "^4.2.4"
  }
}
```

### なぜAstroか

- ほぼ静的なLP（ランディングページ）構成なので、**デフォルトでJSをほとんど吐かないAstro**と相性が良い
- それでいて、お知らせ部分など必要な箇所だけ素のJavaScriptを書ける
- ビルドすると `dist/` に完全な静的ファイルが出力されるので、そのままFirebase Hostingに乗せられる

`astro.config.mjs` はTailwindをViteプラグインとして差し込むだけ。Tailwind v4はこの形で導入できるのが手軽です。

```js
// @ts-check
import { defineConfig } from 'astro/config';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  vite: {
    plugins: [tailwindcss()]
  }
});
```

### Firebase Hosting の設定

ホスティングはFirebaseを選びました。無料枠で十分収まる規模で、`firebase deploy` 一発で更新できる手軽さが決め手です。

```json
{
  "hosting": {
    "public": "dist",
    "ignore": ["firebase.json", "**/.*", "**/node_modules/**"],
    "cleanUrls": true
  }
}
```

ポイントは2つだけ。

- `"public": "dist"` … Astroのビルド成果物ディレクトリをそのまま公開対象に
- `"cleanUrls": true` … `/legal.html` を `/legal` でアクセスできるようにする（URLから拡張子が消える）

デプロイの流れは `npm run build` → `firebase deploy` だけ。サーバー保守がゼロになり、**維持コストは実質ドメイン代のみ** になりました。

## デザインAI、最初は専用ツール → すぐクォーター切れ → Claudeへ

正直、ここが一番伝えたかった部分です。

最初は「Web専用のデザインAI」を使ってデザインを起こそうとしました。確かに見栄えの良いものは出るのですが、**無料枠（クォーター）がすぐに尽きてしまい**、試行錯誤の途中で手が止まってしまいました。

そこで「ダメ元で」と思い、**Claude** にデザインとコーディングを丸ごと頼んでみたところ、これが想像以上に良い仕上がりでした。

具体的に良かった点：

- 「月明かりのような優しさ」というサロンのコンセプトを伝えると、**ダークネイビー × ゴールド** の上品な配色を提案してくれた
- AstroコンポーネントとTailwindのクラスを直接書いてくれるので、**出力をそのままプロジェクトに貼れる**
- グラデーションテキスト、フェードインアニメーション、スクロール追従ヘッダー、モバイルメニューといった「それっぽい演出」も一緒に組み込んでくれた

たとえばヒーローのキャッチコピーはこんな感じで、Tailwindのグラデーションクラスがそのまま使われています。

```html
<h1 class="text-[clamp(2.2rem,10vw,4.5rem)] md:text-7xl font-bold leading-[1.2]">
  大切な家族を、<br/>
  <span class="text-transparent bg-clip-text bg-gradient-to-r
               from-yellow-100 via-yellow-400 to-yellow-100">
    月明かりの優しさで。
  </span>
</h1>
```

「専用ツールでなければ良いデザインは出ない」と思い込んでいたのですが、コンセプトと制約（Astro+Tailwindで出力してほしい等）を具体的に伝えると、汎用的なAIでも実用十分なものが出てくる、というのは収穫でした。

## お知らせ（Topics）を非エンジニアが更新できる仕組み

静的サイトの弱点は「内容を更新するのにビルド＆デプロイが必要」なこと。でも、お知らせ（Topics）は店舗側の人がエンジニアの手を借りずに更新したいはずです。

そこで、**お知らせだけはGoogleスプレッドシートをデータソースにし、ブラウザ側（クライアントサイド）で読み込んで描画する** 構成にしました。これなら店舗の人は **スプレッドシートに行を追記するだけ** で更新できます。

### 仕組み

1. 共有したGoogleスプレッドシートに「日付・タイトル・本文」の列を用意
2. スプレッドシートを「リンクを知っている全員が閲覧可」で共有
3. サイト側からスプレッドシートの **gviz（Google Visualization API）エンドポイント** をfetchしてJSONで取得
4. 取得したデータをJavaScriptでDOMに描画

スプレッドシート側はこんなイメージです。

| 日付 | タイトル | 本文 |
| --- | --- | --- |
| 2025/04/01 | 営業時間変更のお知らせ | 4月より… |
| 2025/04/20 | GW期間の休業について | 5/3〜5/5は… |

### 実装コード

gvizエンドポイントは、JSONの前後に余計な文字列が付いて返ってくるので、その部分を切り落としてからパースするのがコツです。

```js
async function fetchNews() {
  const SPREADSHEET_ID = 'スプレッドシートのID';
  const url = `https://docs.google.com/spreadsheets/d/${SPREADSHEET_ID}/gviz/tq?tqx=out:json`;

  const response = await fetch(url);
  const text = await response.text();

  // 先頭47文字と末尾2文字はgviz特有のラッパー。これを除去してJSONパースする
  const jsonStr = text.substring(47, text.length - 2);
  const data = JSON.parse(jsonStr);

  // 各行を { 日付, タイトル, 本文 } に整形。新しいものを先頭にしたいのでreverse
  const newsItems = data.table.rows.map(row => ({
    date: row.c[0]?.f || row.c[0]?.v || '', // .f は表示用フォーマット済みの値
    title: row.c[1]?.v || '',
    content: row.c[2]?.v || ''
  })).reverse();

  render(newsItems);
}
```

ハマりどころを補足すると：

- **`text.substring(47, length - 2)`** … gvizは `/*O_o*/\ngoogle.visualization.Query.setResponse({...});` のような形で返すため、JSON本体だけを取り出す必要があります
- **`row.c[0]?.f`** … 日付セルは生値（`.v`）がシリアル値や`Date(...)`形式になることがあるので、表示用にフォーマット済みの **`.f`** を優先して使うと安定します
- 取得後は自動スクロール（一定間隔で次のお知らせへ）と、クリックでモーダル全文表示、も付けています

### この方式のメリット・デメリット

メリット：

- 店舗側はスプレッドシートに書き込むだけ。**再ビルド・再デプロイ不要**
- バックエンドもDBも不要。サーバーレスのまま「更新できる箇所」を作れる
- 無料

デメリット・注意点：

- クライアントから直接読みに行くので、**スプレッドシートの中身は実質公開情報**（公開して問題ない内容だけ載せる）
- スプレッドシートを「閲覧可」で共有する必要がある
- gvizは非公式仕様に近いので、将来仕様変更のリスクはゼロではない

身内向け・お知らせ用途であれば、この手軽さは十分に割に合うと判断しました。

## まとめ

- **予約機能をやめた**ことで「動的である必要」が消え、WordPressから静的サイトへ移行
- **Astro + Firebase Hosting** で、維持コストをほぼドメイン代だけに削減
- デザインは専用AIのクォーター切れを機に **Claude** へ。コンセプトと制約を伝えれば実用十分な仕上がり
- 更新したい箇所（お知らせ）だけ **Googleスプレッドシート + クライアントサイドfetch** で、非エンジニアでも運用可能に

「全部を動的にしない。更新が必要な一点だけ外部データソースに逃がす」という割り切りが、小規模サイトでは費用対効果が高いと感じました。同じように個人や知人のお店のサイトを頼まれた人の参考になれば嬉しいです。
